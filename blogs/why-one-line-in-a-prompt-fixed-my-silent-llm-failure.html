Why One Line in a Prompt Fixed My Silent LLM Failure

April 8, 2026 4 min read
LLMs Prompt Engineering Reliability MCP

I was debugging a strange issue in a regional content generation pipeline. On paper, the setup was simple: a script sent a prompt to an LLM Gateway, the model called an MCP tool named get_regional_summary, and then it was supposed to return a structured JSON object with four text sections like living_in_city, population_overview, popular_districts, and current_real_estate_market.

For some places, this worked perfectly. For others, it failed in a way that was hard to reason about. 

Could not parse ai_generated_content for '/de/bayern/landshut' -- using empty strings. Response preview: ''
Skipping '/de/bayern/landshut' -- ai_generated_content incomplete, will retry on next run

At first glance, it looked like either the MCP tool was broken or the place-specific data was missing. But that turned out not to be true.

The Tool Was Fine

I tested the same location, /de/bayern/landshut, directly in Swagger UI against the LLM Gateway. When I used a much simpler prompt that only asked the model to call get_regional_summary and return the exact JSON from the tool, it worked. The tool returned valid location metadata, infrastructure numbers, and market data.

That was the key insight: the failure was not in the tool call. The failure happened after the tool call, during the generation step.

Where the Prompt Broke Down

My original prompt asked the model to produce a polished four-part living guide with a long list of constraints:

  • Use the MCP tool as the source of truth
  • Do not invent facts
  • Mention concrete local areas
  • Include named local entities
  • Describe housing differences
  • Avoid generic phrasing
  • Return strict JSON

That sounds reasonable, but in practice it created a conflict. For a place like Braunschweig, the model apparently had enough confidence to satisfy those requirements and still generate a response. For Landshut, it likely hit a contradiction: it was told not to invent anything, but it was also asked to produce very specific local detail that was not directly available from the tool output.

Instead of giving a cautious fallback answer, it returned nothing.

The Fix Was One Line

The fix was almost embarrassingly small. I added one instruction under ## Edge-Case Handling: 

* Never return an empty response

And suddenly the same location started working.

Why That Helped

The model needed a tie-breaker. Without that instruction, it seems to have treated silence as the safest option when the prompt became too constrained. With that one line, it had permission to degrade gracefully instead of failing silently.

This was a useful reminder that prompt bugs are often not about obvious syntax mistakes. They are often about instruction priority. If a prompt contains competing goals like "be precise," "don't invent," "be detailed," and "follow strict structure," the model may need an explicit rule for what to do when those goals cannot all be satisfied at once.

In my case, "never return an empty response" acted as that rule.

The Bigger Lesson

If you are building structured LLM workflows, especially ones that combine tool use and strict output schemas, always define the fallback behavior. Do not assume the model will improvise a graceful recovery on its own.

  • If you want safe output, say so.
  • If you want complete output, say so.
  • If you want the model to prefer a cautious generic answer over silence, say that explicitly.

Sometimes reliability improves not because the model got smarter, but because the prompt got clearer about failure handling.

Short Version

One tiny prompt change solved a bug that looked like a tool failure, a parsing bug, or a location-specific issue. It was none of those. The model simply needed permission to respond imperfectly instead of not responding at all.

That permission was one line: 

* Never return an empty response
